# The Life Doula School: Claude Code Instructions

## Project summary

This is the complete rebuild of https://thelifedoulaschool.co.uk. Becca (Rebecca Sarah) no longer offers 1:1 services; she has pivoted entirely to teaching a living philosophy called The Life Doula Bridge, themed "Bridging Heaven and Earth", a middle path blending spiritual and practical everyday living, inclusive of all faiths and none, never pushy. Nothing is sold on this site. Its purpose is to inspire, inform, and support her future course and YouTube channel.

The client is minimally available and delivers content non-linearly in waves over weeks; the site must always look complete and absorb content as it arrives. The first design may be rejected by the client or by David, so the architecture makes a full restyle nearly free: everything visual flows from design tokens. David manages all content himself via Claude Code. There is no CMS and there never will be. Hosting is Cloudflare Pages, fully static output, auto-build on push to `main`, preview deployments per branch.

## Standing rules

- **Tokens only.** Never hard-code a colour, font family, or spacing value in a component or page. All visual values come from CSS custom properties defined in `src/styles/tokens.css` and the active theme in `src/styles/themes/`. Stylelint enforces this; do not weaken the rule.
- **Content lives in content collections** (`src/content/`), never inline in components or pages. A redesign must never touch a markdown file, and a content change must never touch a component.
- **UK English** throughout: content, metadata, and code comments that refer to content.
- **No en or em dashes anywhere.** Use commas, colons, semicolons, or rewrite the sentence.
- **SEO checklist on every page** (see §8 of the original brief, mirrored in `docs/seo-checklist.md`): unique title and description (schemas make them required; description 50 to 160 chars), canonical, full OG and Twitter set, correct JSON-LD, breadcrumbs on nested pages, semantic heading order.
- **Definition of done for any task**: `npm run format && npm run lint && npm run check && npm run build` all pass; SEO fields present; `docs/content-status.md` updated; changes reviewed against the token rule; preview deployment link provided to David.
- **No client-side JS without written justification** in a code comment. Current justified scripts: mobile nav toggle, YouTube facade activation.
- **Never use the client's raster images structurally** (heroes, backgrounds, cards). Her "spiritual postcard" images may only appear inline within article body content, framed as content artefacts. Her SmartArt diagrams are specifications only; rebuild as native SVG components with her wording verbatim.
- **Her supplied wording is sacrosanct.** When client copy arrives, replace placeholders verbatim; do not editorialise.
- Keep `docs/content-status.md` and `docs/redirects.md` current with every content change.
- **URL continuity**: no indexed URL may 404. Trailing slashes always. Any page removal requires a 301 in `public/_redirects` and an update to `docs/redirects.md`.
- Git and deploys are handled manually by David; never run git or deployment commands.

## Working with this repo

- `npm run dev` to develop, `npm run test:build` for build plus internal link check.
- Themes: swap the single `@import` in `src/styles/theme.css` between `themes/dawn.css`, `themes/dusk.css`, `themes/garden.css` to restyle the whole site for a demo.
- `npm run contrast:audit` verifies WCAG AA for every token pairing in every theme.
- `npm run og:generate` regenerates `public/og-default.png` from the active theme tokens.
- YouTube data: `src/lib/youtube.ts` fetches the channel RSS at build; last successful data is cached in `src/data/latest-videos.json` (committed). Feed failure never breaks the build.
